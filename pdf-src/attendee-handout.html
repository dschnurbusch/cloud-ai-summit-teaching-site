Your first useful AI workflow

A take-home guide for finding recurring legal work, designing a safe workflow, and testing it without turning your firm into a science fair.

June 25–26, 2026 · Orlando, FL

The 5-part workflow test

1. Trigger

When should the workflow start? Example: new consultation scheduled, client uploads documents, weekly metrics updated.

2. Inputs

What facts or files does it need? If the answer is "everything in our firm," narrow it.

3. Output

What should it produce? Checklist, draft email, summary, dashboard, table, or set of next actions.

4. Reviewer

Who checks the output before anyone relies on it?

5. First test

What fake or impossible example can you use to see how it fails?

30-minute Monday test

  1. Pick one workflow from the summit.
  2. Gather three real examples and one fake/impossible example.
  3. Write the job description before opening the tool.
  4. Run the workflow on the fake example first.
  5. Run it on one real example, but do not send/use the output yet.
  6. Mark every correction a human had to make.
  7. Decide: keep testing, narrow scope, or abandon.

Starter prompts

Connector builder

Build a read-only connector to [SYSTEM]. Identify the auth method, minimum scopes, environment variables, one safe test fetch, error handling, and what to test before adding write permissions.

KPI dashboard starter

Build a local KPI dashboard from [APP/API/CSV]. Start with fake data, keep metric definitions visible, flag stale data, add a narrative summary, and give deployment options: Railway, Tailscale, or local-only.

Skill authoring starter

Create a SKILL.md for [WORKFLOW]. Include when to use it, required inputs, workflow steps, output format, safety rules, common mistakes, and verification checklist.

Scheduled agent starter

Turn [WORKFLOW] into a scheduled agent. Define data collection, tool access, forbidden powers, deduping, report format, silent no-op behavior, and human approval points.

Red flags / green flags

Red flags

  • No reviewer
  • Output goes directly to client
  • Old files with unaudited permissions
  • No source material
  • No owner

Green flags

  • Task happens weekly
  • Output is a draft/checklist/summary
  • Clear reviewer
  • Can test with fake data
  • Success is measurable
